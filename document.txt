Project Overview
Project Overview – TRACE
TRACE (Technical Round Assessment & Coding Ecosystem) is a structured, cohort-based coding assessment platform built exclusively for colleges. It is not a public coding site, not a subscription model, and not a practice marketplace. TRACE is an institutional assessment engine designed to simulate real technical interview preparation within controlled college batches.
The goal of TRACE is to bring discipline, accountability, and measurable improvement into campus coding preparation. Instead of random practice or open platforms, students are enrolled batch-wise into structured Tracks where they solve one curated problem daily. Performance, streaks, rankings, and analytics are automatically calculated, creating a competitive yet controlled technical growth ecosystem.
TRACE is built for internal college cohorts only — no guest users, no external signups, and no public question access.
________________


Core Vision
TRACE is designed to solve three major problems in campus technical preparation:
1. Lack of structured daily consistency.

2. No batch-level performance visibility.

3. No real technical round simulation discipline.

The system enforces a fixed daily problem schedule, streak logic, ranking system, and track-based progression to replicate real hiring preparation timelines.
________________


System Model
TRACE follows a strict hierarchical engine:
College → Batch → Track → Daily Question → Submission → Analytics
Everything inside the platform depends on:
   * College

   * Batch

   * Track

   * Question

   * Submission

Students cannot exist independently outside this structure.

Platform Architecture
TRACE has two major interfaces:
1️⃣ Student Portal (User Side)
Designed for daily coding discipline and measurable improvement.
Key Functionalities
Daily Challenge Engine
Students receive exactly one question per working day based on their batch start date and assigned track. Future questions remain locked.
Structured Roadmap
Track-based progression ensures a predefined sequence of curated problems (DSA, core topics, interview-level questions).
Submission Engine
Students submit code, which is evaluated against test cases. Results are stored with attempt count, accuracy, and timestamps.
Streak & Ranking System
Streak resets if a day is missed. Ranking is dynamically calculated based on:
      * Total Correct

      * Accuracy

      * Submission Speed

Performance Analytics
Students can view:
         * Overall accuracy

         * Topic-wise accuracy

         * Attempt trends

         * Improvement percentage

         * Batch rank

No Distractions
There is no practice mode, no public browsing, no discussion forum — only structured daily execution.
________________

Core Engine Logic
Working Day Calculation
workingDay = floor((currentDate - batch.startDate) / 1 day)
Question Mapping
questionId = track.questionIds[workingDay]
Access Control
If:
                        * Student has no batch → Access denied

                        * Student inactive → Login blocked

                        * College inactive → Login blocked

                        * Working day exceeds track duration → Completed state

Streak Logic
If submission exists for current workingDay → increment
If missed → reset
Rank Logic
Sorted by:
                           1. Total Correct

                           2. Accuracy

                           3. Fastest Average Submission Time

6. SUBMISSION ENGINE
Daily Limits:
                                                                                                   * Max 3 Runs

                                                                                                   * Max 1 Final Submit

Rules:
                                                                                                      * Duplicate submission for same workingDay blocked

                                                                                                      * Validate question belongs to track

                                                                                                      * Validate student belongs to batch

                                                                                                      * Validate workingDay matches question index

                                                                                                      * Submission endpoint must be idempotent

                                                                                                      * Prevent race conditions using atomic DB operations

Judge Flow:
                                                                                                         1. Receive code

                                                                                                         2. Send to Judge

                                                                                                         3. Receive result

                                                                                                         4. Store result

                                                                                                         5. Calculate score

                                                                                                         6. Update streak

                                                                                                         7. Recalculate leaderboard (atomic)

________________


7. SCORING SYSTEM (OUT OF 10)
Score = Accuracy (5) + Efficiency (3) + Discipline (2)
Accuracy (0–5):
Based on % test cases passed.
Efficiency (0–3):
Based on execution_time / time_limit.
Discipline (0–2):
                                                                                                            * Solved in 1 run → 2

                                                                                                            * 2 runs → 1

                                                                                                            * 3 runs → 0

                                                                                                            * Failed → 0

Recalculation Trigger:
                                                                                                               * Only on successful final submission

                                                                                                               * Must be atomic

________________


8. RANKING ALGORITHM
Sort By:
                                                                                                                  1. Total Correct

                                                                                                                  2. Accuracy %

                                                                                                                  3. Average Submission Time

                                                                                                                  4. Earliest Submission Timestamp

Batch-isolated only.
Leaderboard must support pagination.
Target query performance: < 200ms.
